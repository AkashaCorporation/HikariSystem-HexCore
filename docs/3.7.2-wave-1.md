# HexCore v3.7.2 Wave 1

> Draft execution plan
> Date: 2026-03-18
> Scope: First engine-facing hardening wave before `v3.7.2`
> Theme: Wrapper Fidelity + Honest Degradation

---

## 1. Purpose

`Wave 1` exists to fix the gap between:

- what the native engines can already do
- what HexCore actually exposes and trusts at product level

This wave should stay intentionally narrow.

It should not try to solve:

- full IPC refactor
- worker architecture redesign
- deep Remill semantic repairs
- Souper integration

It should solve:

- Capstone async wrapper parity
- LLVM-MC wrapper parity
- degraded-mode honesty
- doctor/capability visibility for those two paths

This is the safest first wave because it improves correctness and operator trust without opening the heaviest runtime risk surface.

---

## 2. Wave Theme

**"The product should stop pretending the wrapper is stronger than it is, and should stop hiding when it is weaker than the native."**

That means:

- preserve engine behavior more faithfully where possible
- explicitly report unsupported/degraded behavior where not possible
- improve the contract seen by headless pipelines and by operators

## 5. Why These Two Engines First

These two engines are the best first wave because:

- they are heavily used by the disassembler
- they affect both interactive and headless flows
- they have lower crash risk than the Unicorn/worker surface
- the current issues are mostly wrapper/product parity issues, not existential engine issues
- improvements here immediately help trust in static and runtime disassembly workflows

---

## 6. Wave 1 Goals

## 6.1 Goal A - Capstone Async Parity

Current issue:

- the async path does not preserve most handle options
- product behavior is narrower than the native behavior
- fallback can become misleading when Capstone is unavailable

Wave 1 goal:

- make async disassembly preserve the intended option set
- document the support boundary for async mode
- improve architecture trust when native disassembly is unavailable

## 6.2 Goal B - LLVM-MC Wrapper Parity

Current issue:

- native exposes more than the TypeScript/product layer uses
- CPU/features are not surfaced in practical product usage
- address semantics are easy to over-assume
- multiple-instruction assembly has an approximation model that is not explicit enough

Wave 1 goal:

- expose the useful knobs
- clarify trust boundaries
- improve patch workflow diagnostics

## 6.3 Goal C - Honest Degradation

Current issue:

- some unsupported or degraded paths can still look superficially “valid”
- this is dangerous for automation and for analyst trust

Wave 1 goal:

- no silent architecture substitution
- no fake-success shape when the engine is missing
- doctor/capabilities should tell the truth about degradation

## 7. Capstone Work Items

## 7.1 Native Wrapper

### Item C1 - Audit current option propagation

Review and classify all options currently accepted by the binding:

- `SYNTAX`
- `DETAIL`
- `SKIPDATA`
- `MODE`
- `UNSIGNED`
- any additional option supported by the binding surface

Desired output:

- explicit list of options supported in sync mode
- explicit list of options supported in async mode
- explicit list of options intentionally unsupported in async mode

### Item C2 - Preserve async-relevant options

Current problem:

- async only tracks `DETAIL`

Wave 1 target:

- propagate the subset of options that materially changes output and is safe to preserve

Minimum candidate set:

- `DETAIL`
- syntax-related options
- skipdata-related options, if the async architecture can safely support them

If an option cannot be supported safely:

- fail explicitly
- or document and surface degradation clearly

### Item C3 - Validate sync vs async parity

Add tests that compare sync and async output under:

- default configuration
- detail mode
- syntax variation where relevant
- any supported skipdata or mode-setting behavior

Success condition:

- parity is a tested property, not an assumption

## 7.2 Product Wrapper

### Item C4 - Expand wrapper support contract

In `hexcore-disassembler`, document and enforce:

- which Capstone features are exposed
- which are not exposed
- what happens when Capstone is unavailable

This should be visible to:

- users
- pipelines
- `doctor`

### Item C5 - Fallback behavior review

Current principle:

- if native disassembly is unavailable, fallback must be safe

Wave 1 rule:

- fallback may be coarse
- fallback may be partial
- fallback must never impersonate a different architecture

Needed review:

- x86/x64 fallback
- ARM32 fallback
- ARM64 fallback
- MIPS or any other currently detected but weakly supported path

If a fallback is not trustworthy:

- mark unsupported/degraded instead of guessing

---

## 8. LLVM-MC Work Items

## 8.1 Native Wrapper

### Item L1 - Trust model audit

Document the real behavior of:

- `assemble`
- `assembleAsync`
- `assembleMultiple`
- address parameter handling
- object-file parsing and code-section extraction

This wave should explicitly state what is:

- exact
- approximate
- best effort

### Item L2 - Clarify address semantics

Current danger:

- callers can assume the `address` parameter means more than it currently does

Wave 1 target:

- make the semantics explicit in code, docs, and diagnostics
- if exact PC-aware assembly is not provided, stop implying it

### Item L3 - Test approximation boundaries

Add tests covering:

- single instruction assembly
- multiple instruction assembly
- CPU-specific assembly
- feature-specific assembly
- cases where byte attribution is approximate

Goal:

- define the trust boundary instead of hand-waving it

## 8.2 Product Wrapper

### Item L4 - Expose CPU and feature selection

Current issue:

- native supports `cpu` and `features`
- wrapper/product path does not meaningfully expose them

Wave 1 target:

- surface CPU/features through the TypeScript wrapper
- make them available for advanced assembly/patch workflows

Even if the UI does not use them immediately, the headless/programmatic path should.

### Item L5 - Expose async assembly usefully

Current issue:

- async exists natively but is not meaningfully used in the product wrapper

Wave 1 target:

- expose `assembleAsync` where it materially improves responsiveness or headless orchestration

### Item L6 - Patch diagnostics improvement

Improve patch result reporting for:

- instruction grows beyond original slot
- alignment-sensitive NOP padding
- approximate byte attribution in multi-instruction mode
- unsupported syntax/CPU/feature combinations

Success condition:

- patch failure is descriptive enough for analysts and testers

---

## 9. Cross-Cutting Product Work

## 9.1 Doctor and Capability Reporting

Wave 1 should extend diagnostics for Capstone and LLVM-MC specifically.

Doctor should report:

- native module present/missing
- prebuild loaded vs dev build loaded
- architecture supported vs unsupported
- degraded fallback active
- wrapper capability narrower than native capability

This can be done incrementally.

It does not need the full `Wave 2` or `Wave 3` runtime model.

## 9.2 Headless Output Honesty

For any headless command depending on Capstone or LLVM-MC:

- engine unavailable should be explicit
- fallback mode should be explicit
- unsupported architecture should be explicit

No ambiguous “worked but actually degraded silently” behavior.

## 10. Testing Plan

## 10.1 Engine Tests

### Capstone testers should cover

- sync/async parity
- detail parity
- option propagation parity for supported options
- architecture-specific regressions in exposed product paths

### LLVM-MC testers should cover

- CPU/features behavior
- async behavior
- address parameter behavior
- patch diagnostics
- approximation disclosure in `assembleMultiple`

## 10.2 Pipeline Testers Should Cover

- headless commands that depend on Capstone
- headless commands that depend on LLVM-MC
- output schema when engines are present
- output schema when engines are missing
- degraded-mode visibility

## 10.3 Integration Test Matrix

Minimum matrix:

- engine available + supported architecture
- engine unavailable
- engine available + unsupported architecture
- fallback active
- fallback disabled or blocked

## 11. CI / Packaging Requirements

For this wave, packaging and build health must confirm:

- standalone engine versions are published and fetchable
- prebuild naming stays compatible with monorepo loader expectations
- monorepo dev path and packaged path both work
- no hidden runtime dependency assumption is introduced

This wave should not be closed until:

- standalone engine artifact exists
- monorepo integration consumes that artifact cleanly
- IDE rebuild passes

## 12. Deliverables

Wave 1 should produce:

- updated standalone repos for Capstone and LLVM-MC
- updated monorepo integration
- tests for parity/trust boundaries
- doctor/capability visibility improvements
- explicit documentation of supported vs degraded behavior

Optional but recommended:

- a short `wave-1 report` after merge summarizing:
  - engine versions adopted
  - regressions found
  - regressions fixed
  - what is intentionally deferred to wave 2 or 3

---

## 13. Exit Criteria

Wave 1 is done when all of the following are true:

- Capstone async option support is intentional and tested
- Capstone fallback behavior is honest
- LLVM-MC wrapper exposes practical native knobs
- LLVM-MC trust boundaries are documented and tested
- no silent architecture substitution remains in the wave-1 touched paths
- doctor can explain engine availability and degradation for these surfaces
- standalone repo updates have been validated in the monorepo dev environment

---

## 14. Explicit Non-Goals

Wave 1 should not expand into:

- Unicorn worker redesign
- JSON-RPC replacement
- SharedArrayBuffer IPC
- Remill semantic restructuring
- Helix pipeline redesign
- Souper bootstrap

If any of these become necessary, they belong to later waves.

## 16. Bottom Line

`Wave 1` should be the cleanest and least controversial wave in the whole train.

It is not trying to be flashy.

It is trying to make two important engines behave more truthfully at product level.

That is exactly the right first step before:

- `Wave 2` on Unicorn/IPC
- `Wave 3` on Remill/Helix precision
- `3.8.0-beta.1` with Souper
