# HexCore v3.7.2 Wave 2 Runtime Findings

> Working note
> Date: 2026-03-20
> Scope: `hexcore-debugger` + `hexcore-unicorn`
> Focus: runtime failures, root causes, validated fixes, and remaining blockers

---

## 1. Why This Note Exists

`Wave 2` stopped being a generic runtime cleanup wave and became a real debugging campaign against:

- worker lifecycle failures
- PE/x64 emulation correctness gaps
- stale host/worker state sync
- weak runtime scaffolding for Windows process startup
- missing import/API behavior that only appears under real binaries

This document records what was actually observed during validation so the next decisions are based on evidence, not memory.

## 3. Validated Fixes

### 3.1 ARM64 Worker Startup Failure

#### Symptom

- `hexcore.debug.emulateHeadless`
- error: `ARM64 worker not started`

#### Root Cause

- ARM64 worker client startup was weaker than the x64/PE worker clients
- ready/error/exit handling was not robust enough
- startup failures were not surfaced clearly

#### Fix

- hardened startup in:
  - `extensions/hexcore-debugger/src/arm64WorkerClient.ts`
- added:
  - startup timeout
  - `lastStartError`
  - better `not started` vs `not ready` distinction
  - system Node fallback strategy

#### Validation

- `Poly` ARM64 started cleanly
- `executionBackend: worker-arm64` appeared in headless output

---

### 3.2 Session Reuse Causing `UC_ERR_MAP`

#### Symptom

- rerunning headless emulation after a previous session
- error: `Invalid memory mapping (UC_ERR_MAP)`

#### Root Cause

- emulation state was not always being cleaned before a fresh start

#### Fix

- `extensions/hexcore-debugger/src/debugEngine.ts`
- `startEmulation()` now resets active runtime state before re-entry

#### Validation

- repeated runs on `Poly` and `wwzRetailEgs.exe` stopped reproducing the duplicate-map failure

---

### 3.3 PE64 PC Truncation in `worker-pe32`

#### Symptom

- `rip` was 64-bit
- `currentAddress` was truncated to 32-bit style output

#### Root Cause

- `extensions/hexcore-debugger/src/pe32Worker.js`
- x86/x64 mode detection used a fragile heuristic
- the worker could read `EIP` instead of `RIP`

#### Fix

- track `currentArch`
- determine x86 mode explicitly using architecture + mode

#### Validation

- `currentAddress` and `rip` became consistent in PE64 repeated runs

---

### 3.4 Headless Register Snapshot Drift

#### Symptom

- headless outputs showed stale register state
- `rip` and `currentAddress` could disagree

#### Root Cause

- register collection mixed sync and async paths incorrectly
- ARM64 was the first visible case, but x64/x86 also needed cleanup

#### Fix

- `extensions/hexcore-debugger/src/debugEngine.ts`
- added `getFullRegistersAsync()`
- aligned async register reads for:
  - ARM64
  - x64
  - x86
- updated headless handlers in:
  - `extensions/hexcore-debugger/src/extension.ts`

#### Validation

- PE64 outputs now keep `rip` aligned with `currentAddress`

---

### 3.5 Snapshot / Restore Metadata Drift

#### Symptom

- restore brought CPU context back
- but `currentAddress` / `instructionsExecuted` remained at post-run values

#### Root Cause

- snapshot/restore path restored native state but not wrapper metadata
- some calls were not awaited end-to-end

#### Fix

- `extensions/hexcore-debugger/src/unicornWrapper.ts`
- `extensions/hexcore-debugger/src/debugEngine.ts`
- `extensions/hexcore-debugger/src/extension.ts`

#### Validation

- `Poly` and `WWZ` restore flows returned:
  - registers
  - `currentAddress`
  - `instructionsExecuted`
  to the same pre-run state

---

### 3.6 Missing `GS_BASE` for PE x64

#### Symptom

- `wwzRetailEgs.exe` failed very early
- stopped around `155` instructions
- PE/x64 code using `gs:[...]` behaved incorrectly

#### Root Cause

- PE loader mapped TEB/PEB but did not actually initialize segment base correctly for x64

#### Fix

- `extensions/hexcore-debugger/src/peLoader.ts`
- set:
  - `gs_base = TEB`
  - `fs_base = TEB` for x86 path

#### Validation

- `wwzRetailEgs.exe` moved from roughly `155` instructions to `250` clean instructions

---

### 3.7 Fault Diagnostics Were Too Blind

#### Symptom

- `faultAddress` was often missing or misleading
- hard to know whether a crash was:
  - fetch fault
  - read fault
  - write fault
  - mapped-region confusion

#### Fix

- added `faultInfo` reporting in:
  - `extensions/hexcore-debugger/src/pe32Worker.js`
  - `extensions/hexcore-debugger/src/x64ElfWorker.js`
  - worker clients
  - `unicornWrapper`
  - `debugEngine`
  - `extension.ts`

#### Value

- made it possible to see:
  - `currentPc`
  - candidate registers
  - whether a candidate was in-range
  - whether it was already mapped
  - whether auto-map was even attempted

---

### 3.8 TEB Runtime Scaffolding Was Too Thin

#### Symptom

- PE/x64 code accessed:
  - `gs:[0x30]`
  - `gs:[0x58]`
- those offsets matter for:
  - `NT_TIB`
  - TLS pointer semantics

#### Fix

- expanded TEB setup in:
  - `extensions/hexcore-debugger/src/peLoader.ts`
- added:
  - `StackBase`
  - `StackLimit`
  - `ThreadLocalStoragePointer`
  - corrected client id field placement
  - mapped `TLS-vector`

#### Validation

- output showed new regions:
  - `TLS-vector`
  - later `TLS-data`

#### Important Note

- this improved realism
- but did not by itself resolve the `WWZ` fault at `318` instructions

---

### 3.9 Minimal Static TLS Support

#### Symptom

- PE images with TLS directory were still receiving effectively empty TLS runtime state

#### Fix

- `extensions/hexcore-debugger/src/peLoader.ts`
- added minimal static TLS setup:
  - parse TLS directory
  - allocate `TLS-data`
  - point TLS vector slot `0`
  - set `_tls_index = 0`

#### Validation

- `TLS-data` became visible in runtime state

#### Important Note

- again, realism improved
- but the `WWZ` crash at `318` still remained unchanged

---

### 3.10 PE Stub Dispatch Was Not Actually Reaching the Worker Correctly

#### Symptom

- `WWZ` kept failing at the same place
- `apiCalls` stayed empty even though the code path clearly hit imported APIs

#### Root Cause

- `extensions/hexcore-debugger/src/pe32WorkerClient.ts`
- `executeBatch(...)` serialized arguments in the wrong order for the worker
- the worker did not receive PE stub range parameters in the shape it expected

#### Fix

- corrected IPC argument order between:
  - `pe32WorkerClient.ts`
  - `pe32Worker.js`

#### Validation

- this was the turning point
- after the fix:
  - `apiCalls` became populated
  - `WideCharToMultiByte` showed up
  - `instructionsRan` reached `500`
  - `terminatedWithError: false`
  - `currentAddress` moved away from the old crash site

---

### 3.11 Missing `WideCharToMultiByte` / `MultiByteToWideChar`

#### Symptom

- inspection of the failing call-site at `0x14131B837` showed a clear `WideCharToMultiByte` shape
- the runtime had no real handler for that conversion path

#### Fix

- `extensions/hexcore-debugger/src/winApiHooks.ts`
- added:
  - `kernel32!WideCharToMultiByte`
  - `kernel32!MultiByteToWideChar`
- also expanded generic argument capture to 8 arguments in `handleCall()`

#### Validation

- `apiCalls` now shows:
  - `kernel32.dll!WideCharToMultiByte`
  - `returnValue: 0x201`

#### Important Note

- this fix mattered
- but it only became visible after the PE stub dispatch IPC bug was corrected

---

### 3.12 Startup Contract Hardening After Stub Dispatch

#### Symptom

Once PE import dispatch was working, the next repeated APIs in `WWZ` were:

- `GetStartupInfoW`
- `_get_wide_winmain_command_line`
- `GetSystemInfo`
- `K32GetProcessMemoryInfo`
- `__stdio_common_vsprintf_s`

The previous behavior was too thin:

- null command line pointers
- empty startup contract
- process memory info returning failure
- CRT stdio formatting returning `0`

#### Fix

- `extensions/hexcore-debugger/src/winApiHooks.ts`
- added or improved:
  - `GetSystemTimeAsFileTime`
  - `GetSystemInfo`
  - `GetNativeSystemInfo`
  - `GetCommandLineA/W`
  - `GetStartupInfoA/W`
  - `_get_wide_winmain_command_line`
  - `K32GetProcessMemoryInfo`
  - `psapi!GetProcessMemoryInfo`
  - `__stdio_common_vsprintf_s`
  - `ucrtbase.dll!__stdio_common_vsprintf_s`

#### Validation

- `_get_wide_winmain_command_line` now returns a stable heap-backed pointer
- `K32GetProcessMemoryInfo` now returns success
- `__stdio_common_vsprintf_s` now returns a non-zero formatted length

## 6. Remaining Known Gaps

These are no longer hypothetical.

They are the most likely next blockers after the current `500`-instruction success point.

### 6.1 `GetStartupInfoW` Is Still Too Thin

- observed in `apiCalls`
- current return path is now minimally valid, but still not semantically rich
- this may become relevant when the binary starts consuming startup flags or standard handles

### 6.2 `_get_wide_winmain_command_line` Is Still Too Thin

- observed in `apiCalls`
- now returns a real heap-backed pointer
- still uses a minimal empty command line, not a realistic process parameter string

### 6.3 `GetSystemInfo` Should Be More Realistic

- observed in `apiCalls`
- currently sufficient to keep startup moving
- may become semantically relevant for thread count, page size, processor info, or allocation policy

### 6.4 PE `ProcessParameters` Is Still Minimal / Missing

- current work improved TEB and TLS
- `PEB.ProcessParameters` is still not modeled at the same level
- many PE/CRT startup paths eventually lean on this

### 6.5 TLS Model Is Still Minimal

- slot `0` model is enough for early progress
- it is not yet a full Windows TLS implementation
- no TLS callback execution
- no broader slot allocation behavior

### 6.6 CRT Formatting Surface Is Still Minimal

- `__stdio_common_vsprintf_s` now exists
- it is intentionally simple
- no full `va_list` / locale / width / precision / wide-format fidelity yet

### 6.7 Fault Address Recovery for Complex Memory Operands Is Still Heuristic

- `faultInfo` is much better now
- but for some instructions the actual effective address is still not derived directly from the operand form

## 7. Working Interpretation

The runtime is now past the phase where it was mostly fighting:

- worker startup
- duplicate maps
- stale snapshots
- broken PE stub dispatch

The next failures are more likely to be:

- missing Win32 / CRT semantics
- weak PE process environment modeling
- incomplete startup object contracts

That is a better class of problem.

It means `Wave 2` is no longer blocked on plumbing alone.

---

## 8. Recommended Next Cuts

### Cut A - Startup Contract Fidelity

Implement or improve:

- `GetStartupInfoW`
- `_get_wide_winmain_command_line`
- minimal `ProcessParameters`

### Cut B - Deeper PE/CRT API Surface

Improve:

- string/encoding helpers
- startup/runtime helper APIs seen in `apiCalls`
- critical section state if the binary starts depending on structure contents

### Cut C - Better Fault Introspection

For later failures:

- derive effective memory address from the actual instruction shape when possible
- not just candidate registers
