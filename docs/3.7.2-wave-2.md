# HexCore v3.7.2 Wave 2

> Draft execution plan
> Date: 2026-03-18
> Scope: Runtime hardening wave before `v3.7.3` or folded into late `v3.7.2`
> Theme: Unicorn Stability + IPC v1.5
> Detailed runtime notes: `docs/Fixes/WAVE2_RUNTIME_FINDINGS.md`

---

## 1. Purpose

`Wave 2` exists to reduce the gap between:

- the current runtime value HexCore already gets from Unicorn
- the architectural cost paid to keep that runtime stable inside Electron/VS Code

This wave is not about adding flashy emulation features.

It is about making the current runtime architecture:

- more predictable
- less chatty
- easier to reason about
- easier to test
- more honest about degraded or worker-only execution paths

This is the first serious runtime wave.

Unlike `Wave 1`, this wave touches the part of the platform most likely to produce:

- crashes
- race conditions
- stale state sync
- hidden performance cliffs

Because of that, this wave must stay strict and disciplined.

## 5. Why Wave 2 Exists

The current code already proves HexCore can get real value from Unicorn.

That is not the issue.

The issue is that the runtime architecture currently depends on:

- multiple worker variants
- frequent sync loops
- manual register copying
- stack mirroring
- best-effort heap/string sync
- arch-specific escape hatches

This is not failure.

It is a sign the product successfully pushed Unicorn into difficult territory.

But it also means the next release train must harden that runtime before larger engine work piles on top.

## 6. Wave 2 Goals

### 6.1 Goal A - IPC v1.5

This wave should implement an intermediate IPC architecture:

- better than current JSON-RPC style chatter
- smaller than a full shared-memory ABI

Target:

- fewer round-trips
- larger useful batches
- more state reads/writes grouped intentionally
- worker-side suppression of pointless events

### 6.2 Goal B - Worker Protocol Alignment

The ARM64, x64 ELF, and PE32 workers should stop feeling like three separate protocol families.

Target:

- common mental model
- common message naming principles
- common error handling structure
- common lifecycle reporting

### 6.3 Goal C - Runtime Honesty

The product must clearly report when it is running in:

- in-process mode
- worker mode
- degraded compatibility mode
- fallback mode

This matters for:

- debugging
- pipeline testing
- performance interpretation
- crash triage

### 6.4 Goal D - Sync Cost Reduction

The current architecture should stop doing work that it already knows is too expensive.

Examples:

- repeated per-register reads where bulk reads exist
- repeated per-register writes where bulk writes exist
- unconditional sync of state that has not changed
- host sync when the next action does not actually need host visibility

## 7. Core Work Items

## 7.1 Protocol Inventory

### Item U1 - Inventory all worker RPC surfaces

Map and compare:

- ARM64 worker
- x64 ELF worker
- PE32 worker

For each one, inventory:

- start/init lifecycle
- memory map/read/write
- register read/write
- batch register support
- context save/restore/free
- hook event transport
- error transport
- dispose/exit behavior

Desired output:

- one normalized protocol table
- one list of divergences
- one list of "safe to unify now" changes

### Item U2 - Standardize message envelope

All worker protocols should share a common structure for:

- request id
- method
- params
- result
- error
- diagnostics

Goal:

- easier clients
- easier tracing
- easier future transport replacement

## 7.2 Register and Memory Sync

### Item U3 - Eliminate avoidable per-register sync loops

Where batch APIs already exist, use them consistently.

This applies especially to:

- worker -> host sync before hook evaluation
- host -> worker sync after hook mutations
- snapshot/restore prep paths

Acceptance:

- no hot path should still perform obvious single-register chatter when a bulk path exists

### Item U4 - Introduce selective sync rules

The runtime should stop syncing on habit.

Implement selective rules such as:

- only sync argument memory when a hook type actually needs it
- only sync stack slices when stack mutation is relevant
- only push register state back when the hook actually changed state

Goal:

- reduce hot-path cost without changing visible behavior

### Item U5 - Tighten deferred write strategy

Review current deferred memory/register write flow and classify:

- safe immediate writes
- writes that must remain deferred
- writes that can be applied in a batch at known boundaries

This should reduce ad hoc sync behavior.

## 7.3 Hook Transport

### Item U6 - Worker-side event filtering

Current architecture sends more hook/event activity upward than necessary.

Wave 2 target:

- suppress events not consumed
- filter high-frequency events near the source
- only emit events that have a consumer or diagnostic value

This is especially relevant if hook event transport currently exists but is mostly ignored.

### Item U7 - Align hook semantics across workers

For each worker family, define:

- what counts as stop-worthy
- what counts as log-only
- what counts as host-visible
- what remains internal

This avoids each architecture inventing its own host expectations.

## 8. Worker-Specific Work

## 8.1 ARM64 Worker

Focus:

- protocol alignment
- batch operations
- consistency with other workers

Questions to resolve:

- should ARM64 adopt the same system-Node strategy as x64 ELF/PE32 where possible
- which sync operations are still unavoidable
- which diagnostics are unique and which should be normalized

## 8.2 x64 ELF Worker

Focus:

- smart-sync cost control
- API hook boundary clarity
- reduced host mirroring when not required

Questions to resolve:

- how much of current Smart Sync is essential
- how much is protective over-sync
- what can be turned into conditional sync based on callsite shape

## 8.3 PE32 Worker

Focus:

- WinAPI stub dispatch efficiency
- x86/x64 split correctness
- runtime consistency after host-side mutation

Questions to resolve:

- where register sync is excessive
- where stack sync is excessive
- what can be merged with the x64 ELF host-sync model

## 9. Product-Level Work

## 9.1 Debugger Headless Contract

All debugger headless commands affected by worker execution should return enough metadata to tell the truth about runtime mode.

Suggested metadata additions:

- `executionMode`
- `workerMode`
- `architecture`
- `degraded`
- `degradationReason`

This is important because the same command may behave differently depending on architecture and worker activation.

## 9.2 Doctor Expansion for Runtime

`pipeline.doctor` should eventually say things like:

- Unicorn native module available
- ARM64 worker path available
- x64 ELF worker path available
- PE32 worker path available
- system Node fallback found/not found
- runtime currently degraded

Wave 2 should at least prepare this visibility.

## 9.3 Logging and Diagnostics Cleanup

Reduce noisy logs and replace them with structured diagnostics where useful.

Target:

- less noise during normal runs
- better signal during failure analysis
- clearer distinction between debug logs and operator-facing warnings

---

## 10. Explicit Non-Goals

This wave should not:

- implement full zero-copy shared memory IPC
- redesign the debugger UX
- solve Remill fidelity
- redesign Helix confidence logic
- introduce new heavy runtime features just because the runtime layer is open

The purpose is stabilization, not opportunistic feature creep.

---

## 11. Testing Plan

## 11.1 Runtime Stress Cases

Testers should cover:

- repeated headless runs without restart
- repeated snapshot/restore cycles
- breakpoint-heavy runs
- high hook density workloads
- API-hook-heavy binaries
- worker disposal and re-entry

## 11.2 Architecture Matrix

At minimum:

- ARM64
- x64 ELF
- PE32 x86
- PE32 x64 if supported by current flow

## 11.3 Failure Modes to Watch

Pipeline and runtime testers should explicitly look for:

- stale host memory during hook evaluation
- stale register state after callback mutation
- repeated worker startup overhead
- worker exit without clear surface error
- memory region migration failures
- duplicate map errors on reruns
- command succeeding with degraded behavior but no diagnostic

## 11.4 Performance Metrics

This wave should collect rough before/after metrics for:

- instruction throughput
- hook-heavy throughput
- average RPC count per emulation window
- batch size effectiveness
- time spent in sync sections

Even rough numbers are better than intuition.

- packaged behavior was checked

## 13. Deliverables

Wave 2 should produce:

- cleaner worker protocol model
- reduced runtime IPC chatter
- stronger runtime metadata in headless output
- better diagnostic truth around worker/degraded modes
- updated tests for repeated-run and worker-heavy cases

Optional but recommended:

- a runtime protocol note or appendix documenting the aligned worker model

---

## 14. Exit Criteria

Wave 2 is done when:

- obvious avoidable per-register chatter is removed from hot paths
- worker protocols are materially more aligned
- runtime mode is visible in affected headless outputs
- repeated-run stability improves measurably
- packaging still works for the runtime paths touched
- no critical worker-mode regression remains open

## 16. Bottom Line

`Wave 2` is the expensive but necessary runtime cleanup wave.

If `Wave 1` makes static engine behavior more honest,
`Wave 2` makes dynamic runtime behavior more sustainable.

This wave should be the bridge between:

- today’s "it works, but the runtime pays a lot for it"

and

- a platform ready for deeper precision work in `Wave 3`
