# HexCore v3.7.2 Wave 3

> Draft execution plan
> Date: 2026-03-18
> Scope: Precision wave before `v3.7.4` and before `3.8.0-beta.1`
> Theme: Remill Fidelity + Helix Confidence

---

## 1. Purpose

`Wave 3` exists to answer the hardest question in the current HexCore stack:

> When Helix output is wrong, how much of that wrongness was created before Helix ever saw the IR?

This is the precision wave.

It is not mainly about:

- adding more decompiler features
- polishing Rellic
- shipping Souper early

It is about:

- measuring Remill fidelity
- reducing semantic shortcuts in the lift stage
- making Helix quality assessment more grounded
- building confidence before the `3.8.0-beta` Souper era

---

## 2. Wave Theme

**"Helix quality starts upstream. If Remill is blurry, Helix will inherit the blur."**

This wave should formalize that as a product rule.

---

## 3. Engines and Modules in Scope

### Included

- `hexcore-remill`
- `hexcore-helix`

### Touched Indirectly

- `hexcore-disassembler`
- fixture suites
- report/diagnostic layers for lift precision

### Explicitly Out of Scope

- new Rellic investment beyond deprecation hygiene
- Souper itself

Recommended standalone bumps:

- `hexcore-remill` `0.1.2 -> 0.1.3`
- `hexcore-helix` `0.6.0 -> 0.6.1` if Helix-side validation or input-handling changes are required

Product version impact:

- this wave should be the last stable-quality precision pass before `3.8.0-beta.1`

---

## 5. Why Wave 3 Exists

Current reality:

- Helix can already produce strong output for a very young engine
- Remill already produces usable IR
- but usable is not the same as trustworthy enough for a Helix-first architecture

Key precision risks already visible in code:

- placeholder conditional behavior
- direct jump target uncertainty
- synthetic terminators
- best-effort CFG repair
- architecture support narrower in product path than in native capability

This wave exists to stop discussing that as intuition and start treating it as measurable engineering reality.

---

## 6. Wave 3 Goals

### 6.1 Goal A - Remill Precision Matrix

This wave should define a precision matrix for:

- x86
- x64
- ARM64

For each fixture, record:

- input bytes
- expected control-flow shape
- lifted block structure
- bytes consumed
- direct call target recovery
- direct branch target recovery
- unresolved branches
- synthetic terminator insertion
- Helix output quality notes

The precision matrix should become a first-class internal artifact.

### 6.2 Goal B - Semantic Shortcut Inventory

Every known semantic shortcut in the lift path should be classified:

- acceptable temporary simplification
- dangerous distortion
- must-fix blocker before `3.8.0-beta.1`

The point is not perfection.

The point is knowing exactly where the product is lying, approximating, or safely simplifying.

### 6.3 Goal C - Helix Confidence Grounding

Helix confidence should not be interpreted in isolation.

This wave should define a rule:

- low-quality input IR can invalidate confidence interpretation

Practical outcome:

- when Helix confidence is high on distorted IR, that case should still be reviewable
- when Helix confidence is low, distinguish whether the problem is:
  - lift-stage distortion
  - decompile-stage limitation
  - unsupported pattern

### 6.4 Goal D - Rellic Exit Hygiene

Rellic is deprecated and scheduled for removal in `3.8.0.`

This wave should reduce the chance that people still treat Rellic as the long-term escape hatch.

That means:

- deprecation clarity
- Helix-first testing
- no new strategic Rellic promises

## 7. Remill Work Items

## 7.1 Native Fidelity Review

### Item R1 - Branch semantics audit

Audit and classify:

- conditional branch condition recovery
- direct jump target recovery
- fallthrough handling
- indirect jump handling
- direct call target injection

Desired output:

- list of exact cases handled correctly
- list of approximate cases
- list of dangerous fallback paths

### Item R2 - Synthetic terminator review

Current lifting strategy may insert synthetic control-flow endings in some cases.

Wave 3 target:

- identify all places where synthetic terminators are inserted
- measure how often that happens on real fixtures
- decide which cases are acceptable scaffolding and which pollute the IR too much

### Item R3 - Architecture support truthfulness

The product path should state clearly:

- what the native can do
- what the product exposes
- what is unsupported at product level even if native constants suggest broader support

This matters because operator expectation should be based on actual supported product paths, not on native surface area alone.

### Item R4 - Version consistency cleanup

Wave 3 should close any obvious version drift between:

- native exported version
- package version
- docs/changelog references

This is small, but important for trust and diagnostics.

## 7.2 Product Integration

### Item R5 - Lift result metadata expansion

The lift result should become more useful for testing and diagnosis.

Suggested additions:

- architecture
- execution notes
- unresolved branch count
- synthetic terminator count
- degraded or approximate flags where applicable

Even if some of these are initially best-effort, surfacing them matters.

### Item R6 - Fixture-first validation path

The monorepo should have a clear test path that validates:

- bytes -> asm
- bytes -> Remill IR
- IR -> Helix output

This should not rely only on ad hoc challenge testing.

Real challenge fixtures are still important, but they should be formalized into repeatable validation.

---

## 8. Helix Work Items

## 8.1 Input IR Validation

### Item H1 - Distinguish bad input from bad decompilation

Helix-side testing should classify failures as:

- malformed/weak input IR
- unsupported legal IR pattern
- actual Helix decompile issue

This prevents the decompiler team from absorbing blame for upstream lift issues.

### Item H2 - Confidence drift tracking

For fixture suites, track:

- confidence before Remill fixes
- confidence after Remill fixes
- readability before/after
- structural correctness before/after

This creates a measurable link between lift fidelity work and Helix output quality.

### Item H3 - Regression fixture bank

Build or curate a bank of representative cases:

- straight-line functions
- branch-heavy functions
- loop-at-entry functions
- call-heavy functions
- table-dispatch or VM-like control flow
- packed/runtime-decrypted handler fragments where available

The point is not quantity alone.

The point is coverage of failure shapes.

---

## 9. Cross-Cutting Precision Work

## 9.1 Precision Taxonomy

This wave should define a shared internal taxonomy:

- `exact`
- `approximate-safe`
- `approximate-risky`
- `unsupported`
- `distorted`

This vocabulary should be used by:

- engine maintainers
- Helix testers
- pipeline testers
- release notes when necessary

## 9.2 Diagnostics for Analysts

The product should progressively move toward exposing when IR is approximate.

This does not have to become noisy UI immediately.

But internally and in headless outputs, the system should be able to say:

- this lift is incomplete
- this branch shape is approximate
- this decompile result depends on repaired CFG

That is more honest than pretending the pipeline is exact.

---

## 10. Testing Plan

## 10.1 Required Fixture Classes

At minimum:

- x86 simple control flow
- x64 direct calls and branches
- x64 branch-heavy logic
- ARM64 conditional flow
- ARM64 loop forms
- one or more runtime-decrypted samples where possible

## 10.2 Review Method

For each important fixture, reviewers should compare:

- disassembled asm
- lifted `.ll`
- Helix pseudo-C

Questions to ask:

- did the CFG survive
- did direct targets survive
- did fallthrough survive
- did the lift inject structure that changes meaning
- did Helix receive enough truthful structure to reason properly

## 10.3 Team Split

Pipeline testers:

- verify headless output shape and metadata

Helix/MLIR testers:

- judge pseudo-C correctness and readability

Engine maintainers:

- trace distortion back to lift logic

Packaging/CI:

- ensure fixture suites remain runnable in the expected environments

---

## 11. Rellic Policy in Wave 3

Rellic should receive only deprecation hygiene in this wave.

Allowed:

- documentation cleanup
- deprecation warnings
- compatibility notes

Not allowed:

- new strategic pass work
- new roadmap promises
- major product investment

The point is to keep the team mentally aligned with the Helix-first future.

---

## 12. CI / Packaging Requirements

Wave 3 should verify:

- Remill semantics are present and load correctly
- Remill standalone artifact is aligned with package versioning
- Helix consumes the expected IR fixtures in CI or pre-CI validation where possible
- no precision fixture quietly rots because of packaging drift

This wave is not closed until:

- fixture validation is repeatable
- not just manually reproduced once

---

## 13. Deliverables

Wave 3 should produce:

- a Remill precision matrix
- a fixture suite or fixture catalog for Remill -> Helix
- a taxonomy for exact vs approximate lift behavior
- clearer Remill result metadata
- a Helix validation note distinguishing upstream vs downstream defects
- Rellic deprecation hygiene updates

Optional but strongly recommended:

- a summary report mapping:
  - top 5 Remill distortions
  - top 5 Helix pain points
  - which of those are blocked by Souper vs independent of Souper

---

## 14. Exit Criteria

Wave 3 is done when:

- a real precision matrix exists
- major known Remill shortcuts are classified
- at least one round of high-value Remill precision fixes lands
- Helix testers can separate input-IR issues from Helix issues more reliably
- Rellic is clearly treated as transitional only
- the team has enough confidence to let `3.8.0-beta.1` begin with Souper afterward

---

## 15. Recommended Owners

Suggested split:

- Remill maintainer:
  - lift fidelity
  - metadata
  - version consistency
- Helix/MLIR tester or maintainer:
  - input IR quality review
  - confidence drift tracking
- Pipeline testers:
  - fixture execution and output consistency
- Docs/release coordination:
  - Rellic deprecation messaging
  - precision taxonomy documentation

---

## 16. Bottom Line

`Wave 3` is where HexCore stops saying "the decompiler is good" and starts proving:

- when the lift is trustworthy
- when the decompile is trustworthy
- when one corrupted the other

That is the right final stable wave before:

- `3.8.0-beta.1`
- Souper integration
- Rellic removal
