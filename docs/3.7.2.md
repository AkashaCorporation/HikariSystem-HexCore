# HexCore v3.7.2

> Draft release plan
> Date: 2026-03-18
> Context: Post-v3.7.1 stabilization, headless-first architecture, Helix ramp-up, Rellic deprecation

---

## 1. Executive Summary

`v3.7.2` should not be treated as a tiny patch release.

The current codebase has already outgrown the old `2.x` roadmap assumptions:

- `hexcore-ai` is no longer the right mental model
- `.hexcore_job.json` plus the headless command surface is the real automation layer
- the product is now operable by CLI, by agents, and by background-open HexCore sessions
- `hexcore-rellic` is deprecated and scheduled for removal in `v3.8.0`
- `hexcore-helix` is the strategic decompiler going forward
- `hexcore-remill` quality now directly affects Helix output quality
- the current IPC model is functional, but architecturally expensive and increasingly visible as a bottleneck

Because of that, `v3.7.2` should be a focused convergence release:

- harden the headless contract
- reduce avoidable IPC overhead
- improve wrapper fidelity where the native engines are stronger than the product layer
- establish measurable precision gates for Remill -> Helix
- prepare a clean path into `v3.8.0`

This release should be positioned as:

> "The release that stabilizes the automation-first HexCore architecture before the 3.8 decompiler transition."

---

## 2. Why v3.7.2 Should Exist

`v3.7.1` shipped two major updates at once:

- Dynamic Intelligence
- XOR Massive Update

That was a strong feature release, but the codebase now shows several pressure points that should be addressed before `v3.8.0`:

- wrapper capability is behind native engine capability
- worker IPC is too chatty in hot paths
- some fallbacks are misleading instead of safely degraded
- Remill is useful, but still takes semantic shortcuts
- Helix quality depends on upstream IR quality more than on Helix itself
- headless mode is now core product surface, not a side feature

If `v3.8.0` is meant to remove Rellic and lean harder on Helix, then `v3.7.2` is the right place to clean the floor first.

Without a `v3.7.2`, `v3.8.0` risks becoming:

- Rellic removal
- Helix growth
- Remill fidelity fixes
- IPC refactor
- wrapper parity
- packaging hardening

all in one release.

That is too much coupling.

---

## 3. Strategic Thesis

`v3.7.2` should have one central thesis:

> HexCore is no longer "a VS Code fork with some extensions".
> HexCore is now an automation-capable reverse engineering platform with an IDE front-end.

This implies four priorities:

1. Headless must be treated as first-class product surface.
2. IPC must be treated as architecture, not just plumbing.
3. Engine wrappers must stop underselling their natives.
4. Helix quality must be judged together with Remill quality.

## 5. Versioning Policy

HexCore should use two different clocks:

1. the public product version
2. the native engine version

These clocks must not be collapsed into one another.

### 5.1 Public Product Version

The product should keep clean SemVer:

- `3.7.2`
- `3.7.3`
- `3.7.4`
- `3.8.0-beta.1`
- `3.8.0-beta.2`
- `3.8.0`

Do not use:

- `3.7.1.1`
- `3.7.1.2`
- any `major.minor.patch.patch` pattern in `package.json`

Reason:

- 4-part numeric versions are not valid SemVer
- Node/npm tooling does not treat `3.7.1.1` as a proper version
- VS Code extension packaging and release ordering become harder to reason about
- it weakens the public release story

### 5.2 Native Engine Version

Each engine should keep its own version rhythm in its standalone repository:

- `hexcore-capstone`
- `hexcore-unicorn`
- `hexcore-llvm-mc`
- `hexcore-remill`
- `hexcore-helix`

This is where delicate patching belongs.

Examples:

- `hexcore-capstone` `1.3.2 -> 1.3.3`
- `hexcore-unicorn` `1.2.1 -> 1.2.2`
- `hexcore-llvm-mc` `1.0.0 -> 1.0.1`
- `hexcore-remill` `0.1.2 -> 0.1.3`
- `hexcore-helix` continues on its own independent cadence

This is the correct place to record:

- wrapper-only fixes
- binding fixes
- linker fixes
- packaging fixes
- prebuild fixes
- engine-specific behavior fixes

without forcing a full public IDE release every single time.

### 5.3 Internal Wave Naming

For planning and team coordination, use wave names instead of fake SemVer:

- `3.7.2-wave-1`
- `3.7.2-wave-2`
- `3.7.2-wave-3`

and/or engine-specific labels:

- `capstone-r1`
- `unicorn-r2`
- `remill-r3`
- `helix-r2`

These are internal planning labels only.

They should appear in:

- draft docs
- Discord coordination
- internal task tracking
- CI discussion

They should not replace the public product version.

### 5.4 Recommended Documentation Pattern

Recommended structure:

- `docs/3.7.2.md` -> macro release plan
- `docs/3.7.2-wave-1.md` -> first engine/integration wave
- `docs/3.7.2-wave-2.md` -> second wave
- `docs/3.7.2-wave-3.md` -> third wave

Each wave document should include:

- engines touched
- standalone repo target versions
- monorepo integration scope
- CI/CD impact
- regression test scope
- exit criteria

## 6. Release Train Proposal

The recommended release train is:

- `3.7.2` -> engine hardening wave 1
- `3.7.3` -> IPC/wrapper wave 2
- `3.7.4` -> Remill/Helix precision wave 3
- `3.8.0-beta.1` -> Souper integration begins
- `3.8.0-beta.2` -> Souper packaging and regression stabilization
- `3.8.0` -> stable Helix-first release

### 6.1 v3.7.2

Theme:

- automation contract hardening
- engine wrapper parity
- IPC reduction groundwork
- Remill precision visibility

This is the first convergence release.

### 6.2 v3.7.3

Theme:

- worker/IPC cleanup follow-through
- more aggressive runtime hardening
- packaging and diagnostics alignment
- broader fixture coverage

This release exists if `3.7.2` uncovers integration fallout.

### 6.3 v3.7.4

Theme:

- final pre-Souper stabilization
- Remill -> Helix confidence work
- deprecation cleanup for Rellic
- release gate cleanup before `3.8.0-beta.1`

This release should be considered the last stable train before the Souper era begins.

### 6.4 Why This Train Is Better Than 3.7.1.1

Because it preserves three things:

- valid tooling semantics
- clean public communication
- precise internal engineering tracking

In short:

- public users see stable SemVer
- engine maintainers get independent patch cadence
- the team still gets granular wave tracking

## 7. Beta Policy for v3.8.0

`v3.8.0` should not be born directly as stable if Souper is entering the architecture.

It should be born as beta.

### 7.1 Why v3.8.0 Must Start as Beta

Souper is not a cosmetic feature.

It touches:

- LLVM IR transformation
- solver integration
- native engine build complexity
- CI/CD packaging
- decompilation quality interpretation
- Helix input quality

That is exactly the kind of change that deserves staged public rollout.

### 7.2 Recommended Beta Sequence

#### `3.8.0-beta.1`

Goal:

- Souper integrated
- native wrapper compiles
- pipeline can invoke it
- output is inspectable

But not yet promised as stable.

#### `3.8.0-beta.2`

Goal:

- CI/CD is stable
- packaging is reproducible
- Helix regressions are understood
- performance and crash behavior are acceptable

#### `3.8.0`

Goal:

- Rellic removed
- Souper path declared stable
- Helix-first stack credible end to end

### 7.3 Stable Criteria for 3.8.0

Do not call `3.8.0` stable unless:

- Souper builds cleanly in CI
- prebuild packaging is reproducible
- Remill -> Souper -> Helix fixtures are green
- no critical regression remains in core headless flows
- documentation no longer describes Rellic as part of the future architecture

#### IDE Integration Maintainers

#### Pipeline Testers

Own:

- `.hexcore_job.json` validation
- headless output stability
- cross-workspace execution behavior
- regression jobs over representative samples

Expected deliverables:

- pass/fail report per wave
- minimal repro jobs when failures appear
- contract drift report if output schema changes unexpectedly

#### Helix / MLIR Decompilation Testers

Own:

- Helix output quality validation
- IR ingestion regressions
- confidence-score drift analysis
- Remill-originated distortion detection

Expected deliverables:

- before/after comparisons on fixed fixtures
- decompilation readability notes
- "Helix issue" vs "Remill issue" split when possible

#### Packaging / CI Monitors

Own:

- standalone build health
- prebuild fetch/install path
- monorepo rebuild health
- installer rebuild validation

Expected deliverables:

- CI status summary
- artifact availability confirmation
- packaging regression warnings

### 8.2 Recommended Workflow Per Engine Change

For delicate native engines, the workflow should remain:

1. change and validate in the monorepo dev environment
2. confirm behavior in the IDE/dev environment
3. migrate the approved change to the standalone repo
4. generate `.node` via CI/CD there
5. pull the resulting version back into the IDE integration flow
6. rebuild and validate the IDE again
7. only then decide whether a public release should move

This is the correct discipline for HexCore.

### 8.3 What Testers Should Watch For

Pipeline testers should explicitly watch:

- headless JSON shape changes
- output file presence and naming
- degraded-mode reporting
- command capability misclassification
- workspace/job routing mistakes
- background-open session issues

Helix/MLIR testers should explicitly watch:

- wrong CFG shape from lifted IR
- impossible branches
- direct call target corruption
- synthetic return artifacts
- confidence drops after IR changes

Packaging testers should explicitly watch:

- missing prebuilds
- naming mismatches
- missing runtime assets
- semantics path failures
- CI/build matrix drift

## 9. What v3.7.2 Should Include

### 9.1 P0 - Must Ship

### P0.1 Headless Contract Hardening

The headless system is now the replacement for the old AI-assistant concept.

Ship in `v3.7.2`:

- tighten the contract for all headless commands around `file`, `quiet`, `output`
- normalize result shapes more aggressively across extensions
- make `pipeline.doctor` stricter and more actionable
- add explicit engine availability and degraded-mode reporting in headless output
- add a "background session ready" diagnostic path for when HexCore is open but not focused

Why:

- agents and CLI users need deterministic shapes
- headless is no longer optional infrastructure
- current capability introspection exists, but should become stronger as an operational contract

Relevant existing surfaces:

- `hexcore.pipeline.listCapabilities`
- `hexcore.pipeline.validateJob`
- `hexcore.pipeline.validateWorkspace`
- `hexcore.pipeline.doctor`
- `.hexcore_job.json` schema and watchers

Relevant backlog tie-ins:

- infrastructure notes in [docs/FEATURE_BACKLOG.md](C:/Users/Mazum/Desktop/vscode-main/docs/FEATURE_BACKLOG.md#L414)

Acceptance:

- every headless command returns a stable JSON envelope
- every degraded path reports why it degraded
- `doctor` distinguishes missing command, missing engine, packaging failure, unsupported arch, and degraded fallback

### P0.2 IPC v1.5 - Reduce Chatter Without Attempting Full v4 Refactor

The current IPC is not good enough for the platform HexCore is becoming.

But `v3.7.2` should not attempt full `SharedArrayBuffer` zero-copy IPC yet.

Instead, `v3.7.2` should ship a transitional IPC improvement layer:

- batch register reads and writes everywhere they still happen per-register in hot paths
- standardize worker RPC payload shapes across ARM64, x64 ELF, and PE32 workers
- reduce worker-host sync frequency for hook paths when state is unchanged
- move more filtering decisions into worker side before sending events upward
- unify memory dump transport to larger chunked binary-oriented messages where possible
- centralize BigInt serialization strategy instead of letting each worker/client pair drift

This is the right `3.7.2` scope.

What should explicitly not happen in `3.7.2`:

- full external `ArrayBuffer` / `SharedArrayBuffer` CPU state exposure
- full native hook shared-memory ABI
- full zero-copy architecture

That is still `v4.0.0` class work.

Relevant backlog item:

- `#31 Zero-Copy IPC Shared Memory (SharedArrayBuffer)` in [docs/FEATURE_BACKLOG.md](C:/Users/Mazum/Desktop/vscode-main/docs/FEATURE_BACKLOG.md#L398)

Recommended framing:

> `v3.7.2` lays the groundwork for zero-copy IPC by collapsing needless JSON-RPC overhead first.

Acceptance:

- no hot path should still do multi-call per-register sync if a block read/write exists
- worker protocol docs are unified
- worker/client code paths for ARM64, x64 ELF, PE32 look architecturally aligned

### P0.3 Remill Precision Audit

This is the most important technical item for Helix quality.

The key question is no longer:

> "Does Remill produce LLVM IR?"

The key question now is:

> "How often is the generated IR trustworthy enough to feed Helix without lying to the analyst?"

Ship in `v3.7.2`:

- formalize a Remill precision audit pass
- add fixtures that compare `asm`, lifted `.ll`, and decompiler output together
- identify and classify semantic loss cases:
  - conditional branch placeholder logic
  - direct jump target discovery failures
  - synthetic terminators
  - broken call target recovery
  - architecture-specific decode gaps

This release should create a precision matrix for:

- x86
- x64
- ARM64

Metrics worth tracking:

- bytes consumed vs expected
- basic block count vs expected
- recovered direct call targets
- recovered direct branch targets
- unresolved branch rate
- forced-true conditional count
- synthetic-return insertion count
- Helix confidence deltas before and after Remill fixes

Acceptance:

- at least one fixture suite exists specifically for Remill -> Helix validation
- unresolved branch behavior is measurable, not anecdotal
- known semantic shortcuts are surfaced in output or diagnostics

### P0.4 Capstone Async Wrapper Parity

Capstone native is stronger than the product path currently using it.

`v3.7.2` should:

- preserve more handle options in async disassembly
- explicitly decide which options are supported in async mode
- surface that support contract in TypeScript wrapper and docs
- stop treating the async path as "detail-only parity"

Why:

- headless and runtime disassembly rely on async paths
- wrapper parity matters more now that disassembly is used both statically and dynamically

Acceptance:

- async path preserves the option set intentionally supported by the wrapper
- unsupported options fail explicitly or document degradation

### P0.5 LLVM-MC Wrapper Parity and Trust Model

`hexcore-llvm-mc` is real, but product integration is narrower than the native API.

`v3.7.2` should:

- expose CPU and feature selection through the TypeScript wrapper
- expose `assembleAsync` at product layer where it helps
- audit `assembleMultiple` trust model and document when per-instruction byte attribution is approximate
- clarify address semantics and avoid implying exact PC-aware assembly if it is not actually honored
- improve patch-time diagnostics when a patch is semantically valid but size-incompatible

Acceptance:

- wrapper exposes `cpu/features` knobs
- approximation boundaries in `assembleMultiple` are explicit
- patch workflow diagnostics are stricter

### P0.6 Remove Misleading Fallbacks

HexCore should degrade gracefully, not incorrectly.

Candidates:

- any architecture fallback that silently switches to another architecture
- any disassembly fallback that emits plausible-looking but wrong output
- any engine-unavailable path that returns success-shaped output without a true result

This matters for trust.

Example class of issue:

- runtime disassembly paths that collapse unsupported arch cases into `x64`

Acceptance:

- unsupported architecture is reported as unsupported
- degraded mode is visibly degraded

### 9.2 P1 - High Value

### P1.1 Runtime Memory Disassembly Follow-Through

This already landed conceptually in `3.7.1`, but `v3.7.2` should treat it as a real platform primitive.

Ship:

- stronger structured output for runtime dumps
- consistent metadata linking dump -> trigger -> breakpoint -> architecture -> engine mode
- better coupling with headless pipelines and report composer
- fixture coverage using runtime-decrypted handlers

This is backlog-adjacent hardening of:

- `#28 Robust Runtime Memory Disassembly`
- `#29 Headless Breakpoint Snapshots & Dumps`

### P1.2 Breakpoint Snapshot Maturity

Current auto-snapshots are useful, but `v3.7.2` can make them much more valuable:

- configurable register sets
- configurable stack window size
- optional memory probe list per breakpoint
- snapshot naming and deterministic output file naming
- better chaining with `onResult`

This makes `.hexcore_job.json` more expressive for staged dynamic workflows.

### P1.3 Pipeline UX for Background-Open Operation

If HexCore is open in the background and controlled externally, the platform should acknowledge that scenario explicitly.

Ship:

- clearer status reporting for workspace/job currently active
- optional lock/session file or status output for active headless session
- more deterministic behavior when multiple workspaces are open
- better "run from workspace X / job Y" visibility in logs and headless output

### P1.4 Doctor Expansion for Engine Fidelity

`pipeline.doctor` should expand from registration health into capability trust:

- engine load status
- worker mode available/unavailable
- degraded fallback risk
- unsupported architecture warnings
- Rellic deprecated warning
- Helix available/unavailable
- Remill semantics present/missing

This should become the operator's first stop before running large jobs.

To keep the release clean, the following should remain out of scope:

Best treatment:

- groundwork only in `v3.7.2`

### 6.2 Souper

Do not mix Souper into `v3.7.2`.

Reason:

- `v3.8.0` already has a clean architectural slot for it
- Remill fidelity and Helix quality should be stabilized first

### 6.3 New Experimental Decompiler Surface

Do not add new Rellic-facing promises.

Reason:

- Rellic is leaving in `v3.8.0`
- every hour spent polishing Rellic beyond deprecation support is stealing time from Helix and Remill quality

### 6.4 Symbolic Execution

Keep backlog `#32` out entirely.

## 13. Acceptance Gates

`v3.7.2` should not ship without explicit gates.

### Gate 1 - Headless Contract

- every headless command has deterministic JSON output
- every degraded path is explicit
- all templates and docs use the current architecture vocabulary

### Gate 2 - IPC Reduction

- no obvious hot path still performs avoidable per-register RPC loops
- worker protocols are aligned
- worker-host sync volume is measurably lower on representative workloads

### Gate 3 - Remill -> Helix Precision

- fixture suite exists
- known semantic shortcuts are measured
- at least one round of fixes lands for high-impact precision loss

### Gate 4 - Wrapper Fidelity

- Capstone async parity improved
- LLVM-MC wrapper exposes practical native knobs
- misleading architecture fallbacks removed

### Gate 5 - Packaging / Environment

- pipeline doctor catches missing semantics / missing prebuild / degraded native load
- installer/build remains green
- no new fragile runtime dependency assumptions

## 16. Recommended Release Decision

Yes, `v3.7.2` is justified.

Not because `v3.7.1` was weak.

It is justified because `v3.7.1` proved HexCore is now operating at a scale where architecture debt becomes product debt very quickly.

My recommendation is:

- do `v3.7.2`
- make it a serious stabilization release
- pull selected backlog items into it, especially around runtime dumps and IPC groundwork
- avoid overloading it with Souper or major new decompiler promises
- use it to make `v3.8.0` smaller, cleaner, and more credible
