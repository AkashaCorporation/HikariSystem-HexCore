# HexCore v3.7.3

> Draft release plan
> Date: 2026-03-22
> Context: Post-v3.7.2 breakout, Helix now viable, Remill no longer the obvious weak link, trust and confidence now matter more than raw feature count

---

## 1. Executive Summary

`v3.7.3` should not try to out-flash `v3.7.2`.

`v3.7.2` already delivered the important architectural jump:

- Capstone and LLVM-MC wrappers became more honest and more capable
- Unicorn runtime stopped failing early on real targets
- headless debugger workflows became materially more reliable
- Remill received its first serious fidelity upgrades
- Helix stopped behaving like a fragile experiment and started producing usable output
- real-world operator results improved sharply on both malware-style targets and AAA game binaries

That means `v3.7.3` should do something different:

> turn impressive output into trustworthy output

This release should be the **Decompiler Trust Release**.

Its purpose is to make HexCore better at answering:

- when the decompiler output is solid
- when it is usable but approximate
- when it is bluffing
- whether the distortion came from Remill, from Helix, or from both

`v3.7.3` should be positioned as:

> "The release that teaches HexCore when to trust its own decompiler stack."

---

## 2. Why v3.7.3 Should Exist

`v3.7.2` proved something important:

- the stack is no longer just interesting in demos
- it can help an external analyst or agent solve real tasks
- it can produce actionable game-analysis and trainer strategies that older versions could not

That is exactly why `v3.7.3` is necessary.

Once the output becomes useful enough to act on, the next risk is no longer:

- crash
- missing prebuild
- worker startup

The next risk is:

- overconfidence
- hidden approximation
- misleading pseudo-C
- a clean-looking answer built on distorted IR

`v3.7.3` exists to reduce that risk before:

- `3.8.0-beta.1`
- Souper-era work
- Helix-first public messaging hardens further

---

## 3. Strategic Thesis

`v3.7.3` should have one central thesis:

> HexCore's decompiler pipeline should become reviewable, not just productive.

This implies four priorities:

1. Confidence must be grounded in evidence, not just successful emission.
2. Remill and Helix defects must be separable in practice.
3. Fixtures must become first-class engineering artifacts.
4. Output honesty must improve even when readability cannot yet improve.

## 6. What v3.7.2 Already Proved

Before planning `v3.7.3`, the project should explicitly acknowledge what changed in `v3.7.2`:

- Helix output is now viable often enough to be compared against IDA instead of only against itself
- Remill IR quality improved materially through:
  - branch lowering fixes
  - `%NEXT_PC` propagation fixes
  - SSA naming cleanup
  - call fall-through boundary fixes
  - LLVM pass pipeline cleanup
  - boundary detection / truncation controls
- headless analysis pipelines can now produce complete reports over difficult PE64 targets
- real use by external agents is now a meaningful signal source

That means `v3.7.3` should not ask:

> "Can HexCore decompile at all?"

It should ask:

> "How do we know when HexCore is telling the truth?"

## 7. Core Goals

### 7.1 Goal A - Confidence Grounding

Current issue:

- Helix can emit `100% (High)` on outputs that are still clearly low-level, partial, or awkward
- the confidence model risks overstating certainty on emit-success rather than semantic quality

`v3.7.3` target:

- make confidence less naive
- reduce false-high-confidence cases
- factor in structural warnings, unresolved calls, raw opcode leakage, fallback emission, and CFG repairs

Success condition:

- confidence becomes more conservative on weak cases
- confidence remains strong on cases that really are strong

### 7.2 Goal B - Fixture-First Validation

Current issue:

- the best evidence is coming from ad hoc manual testing
- that is useful, but too easy to lose or misremember

`v3.7.3` target:

- formalize a fixture bank covering:
  - malware-like binaries
  - AAA game binaries
  - simple control-flow functions
  - medium complexity functions
  - ugly CFG functions
  - vtable-heavy or RTTI-heavy functions

Each important fixture should preserve:

- asm/disasm
- lifted `.ll`
- Helix pseudo-C
- reviewer notes
- classification of whether the issue is Remill, Helix, or mixed

### 7.3 Goal C - Issue Surfacing

Current issue:

- the stack can still be "usable" while hiding where it is approximate

`v3.7.3` target:

- surface more decompilation caveats directly in results
- examples:
  - unresolved call target
  - repaired CFG
  - memory-source compare/test reconstruction
  - raw SIMD or mnemonic fallback
  - truncated lift
  - approximate target reconstruction

This is not about making output uglier.

It is about making it honest.

### 7.4 Goal D - Remill/Helix Separation

Current issue:

- users still have to guess whether wrong output came from lift or decompile

`v3.7.3` target:

- make the review workflow explicitly compare:
  - ASM
  - Remill IR
  - Helix output
- document a repeatable way to classify:
  - `lift issue`
  - `decompile issue`
  - `shared ambiguity`

This should become a team habit, not a one-off debugging trick.

### 7.5 Goal E - Emitter Cleanup Where It Pays Off

Current issue:

- even when Helix is semantically close, emitter roughness can make output look less mature than it is

`v3.7.3` target:

- improve high-value emission cases:
  - raw mnemonic leakage
  - ugly indirect call placeholders
  - excessive label noise
  - overly noisy parentheses or casts
  - obviously synthetic expressions

This should focus on the cases analysts hit repeatedly, not on cosmetic perfection.

## 8. Remill Work Items

### Item R1 - Lift Metadata Expansion

Expand lift metadata so analysis and tooling can answer:

- was the lift truncated
- why it was truncated
- how many blocks were produced
- how many direct call targets were recovered
- whether CFG repair was applied
- whether approximation or fallback paths were used

Suggested result additions:

- `truncated`
- `truncationReason`
- `nextAddress`
- `directCallTargets`
- `unresolvedBranches`
- `syntheticTerminators`
- `approximationFlags`

### Item R2 - Precision Taxonomy

Introduce a shared taxonomy for lift quality:

- `exact`
- `approximate-safe`
- `approximate-risky`
- `unsupported`
- `distorted`

This vocabulary should appear in:

- internal docs
- fixture reviews
- optional metadata in headless lift results

### Item R3 - Chunk / Boundary Truthfulness

The project should be more explicit when a decompile target is:

- a recognized function boundary
- a lifted code window
- a truncated chunk

This matters because Helix quality is harder to judge fairly on arbitrary windows.

### Item R4 - Regression Cases for Recent Fixes

Lock in regression coverage for:

- conditional branch lowering
- `%NEXT_PC` propagation
- call fall-through boundaries
- SSA naming
- optimization pass cleanup

`v3.7.3` should make sure those `v3.7.2` gains do not quietly rot.

---

## 9. Helix Work Items

### Item H1 - Confidence Model Review

Review why obviously rough outputs can still land at `100% (High)`.

Candidate negative signals:

- unresolved call target
- raw mnemonic fallback emitted
- excessive labels or irreducible leftovers
- trivial or suspiciously empty body
- low variable provenance confidence
- synthetic CFG repairs consumed upstream

### Item H2 - Issue Reporting Pass

Improve the `Issues:` section so it captures more than unresolved call targets.

Candidate issues:

- approximate control flow
- raw semantic fallback emitted
- possible truncated source region
- vector/scalar lowering incomplete
- unresolved object provenance

### Item H3 - Emitter Roughness Pass

Target the highest-value emitter pain points:

- indirect call rendering
- raw mnemonic stubs in pseudo-C
- unhelpful synthetic temporaries
- noisy label layout
- obvious memory compare/load confusion when enough signal exists to do better

### Item H4 - Fixture Review Notes

For each major fixture, store:

- what Helix got right
- what Helix got wrong
- whether the wrongness was already present in the IR

This prevents Helix maintainers from absorbing upstream blame unfairly.

## 10. Validation Model

### 10.1 Three-Layer Review

Every important decompile case should be reviewed as:

1. `ASM`
2. `LLVM IR`
3. `Helix C`

Questions:

- did the branch shape survive
- did the calls survive
- did the memory semantics survive
- did the emitter preserve the important meaning
- did confidence overstate the result

### 10.2 External Comparison

Where possible, compare against:

- IDA output
- verified runtime behavior
- working trainer patches
- successful exploitation / challenge solution evidence

The point is not to clone IDA.

The point is to have an external anchor for correctness.

### 10.3 Representative Case Classes

At minimum:

- one malware-like branch-heavy sample
- one AAA PE64 game sample
- one vtable-heavy sample
- one simple "clean win" function
- one ugly chunk/window case
- one SIMD or float-heavy function

- a `docs/3.7.3.md` release plan
- a formal fixture bank or fixture catalog
- confidence-grounding rules
- clearer Remill lift metadata
- stronger Helix `Issues:` reporting
- a short comparison methodology doc:
  - `ASM -> IR -> Helix`

Optional but recommended:

- one or more case studies from:
  - malware analysis
  - AAA trainer analysis

These are high-value credibility assets.

---

## 12. Exit Criteria

`v3.7.3` is done when:

- confidence is meaningfully less naive
- at least one meaningful fixture bank exists
- at least one round of Remill metadata improvements lands
- at least one round of Helix issue surfacing lands
- recent `v3.7.2` Remill fixes are regression-protected
- reviewers can classify a case as Remill vs Helix with less guesswork

---

## 13. Relationship to v3.7.4 and v3.8.0

### 13.1 Recommended Path

The best release train from here is:

- `3.7.3` -> decompiler trust and signal
- `3.7.4` -> optional polish / fallout release only if needed
- `3.8.0-beta.1` -> Helix-first beta, larger decompiler architecture work begins

### 13.2 What Should Stay Out of v3.7.3

Do not overload it with:

- full Souper integration
- Rellic revitalization
- full IPC `#31`
- broad new debugger features
- a rushed `3.8.0` stable narrative

`v3.7.3` should stay disciplined.

Its job is to make the current decompiler stack more trustworthy.

---

## 14. Recommended Changelog Shape

If this plan is followed, the `v3.7.3` changelog should roughly read like this:

### Added

- decompiler fixture validation workflow
- stronger Remill lift metadata
- expanded Helix issue reporting

### Changed

- Helix confidence scoring is more conservative and better grounded
- decompiler docs now distinguish function lifts from arbitrary code-window lifts
- analyst workflow now formalizes `ASM -> IR -> Helix` review

### Fixed

- false-high-confidence decompile cases
- misleading pseudo-C around indirect calls and fallback emission

## 15. Bottom Line

`v3.7.2` proved HexCore can already win on hard targets.

`v3.7.3` should prove HexCore knows when to trust those wins.

That is the right release before:

- `3.8.0-beta.1`
- Helix-first messaging hardens further
- Souper-era work starts entering the architecture
