# HexCore v3.8.0 Beta 1

> Draft execution plan
> Date: 2026-03-19
> Scope: First public beta for the Souper era
> Theme: Souper Bootstrap + Helix-First Transition

---

## 1. Executive Summary

`v3.8.0-beta.1` should not be treated as a preview in the loose sense.

It should be treated as a real beta:

- public enough to validate the next architecture
- constrained enough to avoid pretending it is already stable

This release exists to introduce the first stable-shaped version of the new path:

`machine code -> Remill -> Souper -> Helix`

while explicitly keeping the right to change:

- IR optimization strategy
- packaging details
- performance behavior
- failure diagnostics
- beta gating rules

This is the release where HexCore stops talking about Souper as roadmap and starts shipping it as architecture.

---

## 2. Why Beta 1 Must Exist

Souper is not like adding:

- one more command
- one more analyzer
- one more headless wrapper

Souper changes the middle of the reverse pipeline.

It affects:

- LLVM IR shape
- optimization assumptions
- solver/runtime dependencies
- CI/CD complexity
- Helix input quality
- analyst trust in decompiled output

That is exactly the kind of change that should not jump directly from branch to stable.

`Beta 1` exists to answer:

- can the wrapper compile and package reliably
- can the monorepo consume the engine cleanly
- can the pipeline invoke the optimizer deterministically
- does the optimized IR help Helix often enough to justify the cost
- what new regressions appear once Souper is inserted in the middle

---

## 3. Release Theme

**"The optimization layer becomes real."**

Recommended public framing:

> `v3.8.0-beta.1` is the first beta release of HexCore's next-generation lift-to-decompile pipeline. It introduces the initial Souper optimization layer between Remill and Helix, starts the Helix-first transition in earnest, and opens the architecture for coordinated testing before stable `v3.8.0`.

---

## 4. Primary Goals

### 4.1 Goal A - Souper Exists as a Real Engine

This beta is successful only if Souper is no longer “planned”.

That means:

- native engine exists
- wrapper exists
- monorepo can load it
- pipeline can call it
- diagnostics can explain when it is missing or degraded

### 4.2 Goal B - Souper Can Be Inserted Without Breaking the World

This beta should prove:

- the optimization stage can be inserted between Remill and Helix
- the product still behaves coherently when Souper is absent, unavailable, or disabled
- performance and packaging impact are understood

### 4.3 Goal C - Helix-First Path Becomes the Official Mainline

`v3.8.0-beta.1` should make the transition unmistakable:

- Helix is the primary decompiler path
- Rellic is no longer part of the forward architecture
- Souper is introduced to improve the Helix path, not to revive Rellic

### 4.4 Goal D - Stable Criteria Are Measurable

This beta should define what must be true before `v3.8.0` can be called stable.

That is one of the main reasons to do the beta in the first place.

## 7. Souper Work Items

## 7.1 Native Engine Bootstrap

### Item S1 - Create standalone engine

Required:

- standalone repo layout
- build instructions
- dependency strategy
- wrapper surface
- CI/CD artifact generation

Core rule:

- the engine must fit the same operational discipline as the other native engines

### Item S2 - Define minimal native API

At minimum, the native layer should answer:

- accept LLVM IR text
- return optimized LLVM IR text
- report optimization failure clearly
- expose version info
- expose availability diagnostics

Do not over-design the first API.

The beta needs a usable surface, not an overbuilt one.

### Item S3 - Build and packaging sanity

Before the beta can be announced:

- the engine must build reproducibly
- CI/CD must produce artifacts
- monorepo integration must load the artifacts in dev and packaged contexts

This is not optional.

---

## 8. Product Integration Work

## 8.1 Pipeline Integration

### Item P1 - Insert Souper into the official optimization chain

The product path should become:

- no optimizer
- LLVM-passes only
- Souper optimizer

Suggested control model:

- `optimizerStep: 'none' | 'llvm-passes' | 'souper'`

This should be treated as:

- product-level contract
- pipeline-level contract
- diagnostic-level contract

### Item P2 - Headless path support

The beta should support Souper from headless flows.

That means:

- direct optimizer invocation if useful
- or integration through existing decompile/lift pipeline commands

Either is acceptable, but the behavior must be deterministic.

### Item P3 - Doctor visibility

`pipeline.doctor` should eventually report:

- Souper present/missing
- Souper usable/unusable
- optimizer chain available/unavailable
- degraded decompilation path if Souper is requested but missing

`Beta 1` should at least start that visibility.

---

## 9. Remill and Helix Work During Beta 1

## 9.1 Remill Side

This beta may require targeted Remill work if Souper exposes lift-stage weaknesses more clearly.

Allowed work:

- metadata improvements
- integration stability fixes
- small precision or compatibility fixes required for optimizer path

Not the goal:

- large-scale semantic rewrite in the same beta

## 9.2 Helix Side

Helix work in `Beta 1` should focus on:

- consuming Souper-optimized IR reliably
- identifying optimizer-induced regressions
- comparing readability and confidence before/after Souper

Important:

Helix should not be forced to “hide” upstream distortions.

If Souper reveals an upstream problem, that should be surfaced, not smoothed over.

---

## 10. Rellic Policy for Beta 1

`Beta 1` is not a Rellic revival.

Rellic policy in this beta:

- remains deprecated
- no new strategic investment
- can remain temporarily available only if removal is not yet finished
- all forward-looking messaging must be Helix-first

If a compatibility bridge is needed during beta:

- keep it quiet
- keep it transitional
- do not let docs imply dual long-term decompiler strategy

## 11. Testing Plan

## 11.1 What Beta Testers Need to Answer

The beta test cycle should answer:

- does Souper actually improve the IR on real fixtures
- does it preserve correctness often enough
- where does it regress Helix readability
- what is the packaging cost
- what is the runtime cost
- what are the failure modes when the optimizer is unavailable

## 11.2 Fixture Categories

Recommended fixture classes:

- straight-line x86/x64 functions
- branch-heavy x64 functions
- ARM64 branch and loop functions
- obfuscated arithmetic-heavy functions
- VM-like control-flow samples
- runtime-decrypted handler fragments if available

## 11.3 Comparison Matrix

For each representative fixture, compare:

- Remill IR
- Remill + LLVM passes
- Remill + Souper
- Helix output from each variant

Record:

- correctness
- readability
- confidence
- simplification quality
- regressions

## 11.4 Tester Split

Pipeline testers:

- headless contract
- output consistency
- degraded-mode behavior

Helix / MLIR testers:

- decompile quality before/after Souper
- confidence drift
- structural regressions

Engine / CI testers:

- build reproducibility
- artifact loading
- performance changes

---

## 12. CI / Packaging Requirements

`Beta 1` should not ship without:

- working standalone engine build
- working artifact publication
- monorepo fetch/install path
- packaged load validation
- clear diagnostics for missing optimizer engine

This is especially important because heavy native engines often fail first in packaging, not in local dev.

---

## 13. Beta 1 Exit Criteria

`3.8.0-beta.1` is done when:

- Souper engine exists and loads
- product can route through Souper intentionally
- headless/pipeline behavior is deterministic
- at least one representative fixture set has been reviewed before/after Souper
- no critical packaging blocker remains
- team can explain the top regressions introduced by the optimizer path

---

## 14. What Must Happen Before Beta 2

`Beta 1` should produce a concrete list for `Beta 2`:

- build and packaging pain points
- optimizer correctness edge cases
- Helix regressions
- performance cliffs
- missing diagnostics
- docs and template updates still required

`Beta 2` should be the release where these are reduced enough to start talking about stable.

---

## 15. Stable Criteria for v3.8.0

Do not call `3.8.0` stable unless:

- Souper build pipeline is reproducible
- optimized IR path is operational in the main pipeline
- Helix regressions are understood and acceptable
- no critical missing-engine or degraded-mode path is misleading
- Rellic is effectively removed from the forward architecture
- the team is comfortable telling users the Souper path is real, not experimental theater

## 17. Bottom Line

`v3.8.0-beta.1` should be the release where HexCore finally says:

- Souper is now part of the architecture
- Helix is the main decompiler future
- Rellic is on the way out

but also says:

- we are not pretending this path is stable yet

That honesty is exactly what makes the beta worth doing.
