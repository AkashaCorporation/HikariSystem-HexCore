## Executive Summary

Durante a análise do desafio Callfuscated, a HexCore demonstrou capacidades sólidas em análise estática, mas encontrou limitações críticas na emulação de binários com VMs (Virtual Machines) complexas. Este relatório documenta os achados, compara a emulação HexCore-Unicorn vs Unicorn nativa, e propõe melhorias específicas.

## Challenge Overview

### Binary Characteristics
- **File Type:** ELF 64-bit LSB executable
- **Architecture:** x86-64
- **Obfuscation:** Heavy call-based obfuscation (callfuscation)
  - ~2833 instruções no main, sendo ~66% junk (call/pop r8 pairs)
  - ~941 instruções reais após filtragem
- **Protection:** VM-based validation logic
  - 10 opcodes VM (dispatcher com `cmp eax, 1` até `cmp eax, 0xa`)
  - 192 chamadas `rand()` com seed determinístico (1337 / 0x539)
  - Validação complexa baseada em PRNG glibc

### Key Findings
1. **srand seed:** 1337 (0x539) - determinístico
2. **rand() calls:** 192 total durante validação
3. **VM dispatcher:** 0x4096AB (jump table com 10 opcodes)
4. **Arrays VM:**
   - Operand stack: `[rbp+rax*4-0x950]`
   - VM program/data: `[rbp+rax*4-0x18f0]`
5. **Validation result:** "Incorrect flag. Try again" ou "Correct flag!"

**Error:**

**Root Cause Analysis:**
- RIP = 0x40d084 (dentro da .rodata section, não-executável)
- A VM calculou um jump incorreto ou a HexCore não mapeou a .rodata como executável
- Comparação com Unicorn nativa: **não apresentou este erro**

**Memory Mapping Comparison:**

| Segment | VAddr | Size | Flags | HexCore Mapping | Native Unicorn |
|---------|-------|------|-------|-----------------|----------------|
| .text | 0x401000 | 0xB5D5 | R-X | ✅ Correto | ✅ Correto |
| .rodata | 0x40D000 | 0x130 | R-- | ❌ R-- only | ✅ RWX (permissive) |
| .data | 0x40E000 | 0x2000 | RW- | ✅ Correto | ✅ Correto |

**Recommendation:**
- Adicionar opção `permissiveMemoryMapping: true` no `emulateFullHeadless`
- Quando ativado, mapear TODOS os segmentos como RWX para VMs self-modifying
- Manter o padrão atual (seguro) como default

#### 2. Rellic Decompiler - Defeated by Obfuscation

**Output:**

**Issue:**

**Recommendation:**

  1. Dead code elimination (DCE)
  2. Constant folding
  3. Junk instruction filtering (call/pop, add/sub zero, etc.)
- Adicionar flag `optimizeIR: true` no `hexcore.rellic.decompile`

  - glibc (Linux)

#### 1.2 Implement glibc PRNG
```typescript
class GlibcPRNG {
  private state: number[] = new Array(344);
  private idx: number = 344;
  
  constructor(seed: number) {
    this.state[0] = seed;
    for (let i = 1; i < 31; i++) {
      this.state[i] = (16807 * this.state[i-1]) % 2147483647;
    }
    // ... (full implementation)
  }
  
  rand(): number {
    if (this.idx >= 344) {
      for (let i = 0; i < 344; i++) {
        this.state[i] = this.state[(i + 31) % 344] + this.state[(i + 3) % 344];
      }
      this.idx = 0;
    }
    const result = this.state[this.idx++];
    return (result >> 1) & 0x7FFFFFFF;
  }
}
```

#### 2.2 Junk Instruction Filtering
```typescript
function filterJunkInstructions(instructions: Instruction[]): Instruction[] {
  const filtered: Instruction[] = [];
  
  for (let i = 0; i < instructions.length; i++) {
    const curr = instructions[i];
    const next = instructions[i + 1];
    
    // Filter call/pop r8 pairs
    if (curr.mnemonic === 'call' && next?.mnemonic === 'pop' && next?.op_str === 'r8') {
      i++; // Skip both
      continue;
    }
    
    // Filter add/sub zero
    if ((curr.mnemonic === 'add' || curr.mnemonic === 'sub') && 
        curr.op_str.includes(', 0')) {
      continue;
    }
    
    filtered.push(curr);
  }
  
  return filtered;
}
```

**Benefit:** Análise mais clara de binários obfuscados
